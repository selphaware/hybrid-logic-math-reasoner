## Finding 1 — Underdetermined query (renderer crash on universal fact)

**Severity:** Medium. Crashes `manual_solve` with `RenderError` rather than
returning a meaningful result. No soundness impact (the error surfaces before
any proof is produced).

**Minimised failing example (Hypothesis-shrunk):**

```
KB:    p_1: p(X).        # universal fact — no body, one clause variable
Query: ?- p(?A).
```

**Root cause:**

When `p(?A)` resolves against `p(X).`, the SLD renames `X → ?X_1` (a fresh
meta) and unifies: the accumulated substitution becomes `{?A: Meta("?X_1")}`.
Because `p(X).` has no body literals, `?X_1` is never further bound to a
concrete term. After `_saturate`, the substitution remains
`{?A: Meta("?X_1")}` — the chain `?A → ?X_1` never terminates at a ground
value.

The renderer then attempts `forallE` for clause variable `X`, calls
`apply_to_term(sat, Meta("?X_1"))`, gets `Meta("?X_1")` back (unsaturated),
and raises:

```
RenderError: unsaturated meta in forallE term for var 'X': Meta(name='?X_1')
```

**Why this is architecturally correct behaviour to surface:**

In HLMR's M1 model, a proof must be fully ground — the kernel rejects any
proof containing a `Meta` term. For the query `?- p(?A).` with `p(X).`, the
meta `?A` is underdetermined: any ground value satisfies the clause. There is
no unique ground witness to produce, so there is no valid M1 proof.

The correct M1 response is to signal "underdetermined" to the user rather
than crash.

**Fix applied (provisional M1):**

`src/hlmr/solve/__init__.py` was updated to detect underdetermined queries:
after SLD succeeds, before calling `render()`, it saturates the substitution
and checks whether any query meta still resolves to a `Meta` node. If so, it
returns `(sat, None)` instead of crashing — a new `(Substitution, None)` shape
distinct from `None` (failure) and `(Substitution, Proof)` (success).

The REPL (`src/hlmr/repl/interactive.py`) prints:

```
Goal resolved but no ground witness — query is underdetermined.
```

**Open question for M2 (see `prd_milestone_2.md` §9/§15):**

Should the M2 `Underdetermined` outcome generalise to cover this pure-SLD
case (returning a partially-bound substitution alongside `Underdetermined`)?
Or should the renderer learn to emit a universally-quantified proof for
universal-fact clauses?  The M2 dispatcher design doc (`DISPATCH_DESIGN.md`)
must address this.

## Finding 2 — Depth-500 occurs check hits Python's call stack

**Severity:** Low. Python implementation limit; no logic error. No soundness
impact (the unifier correctly rejects cycles at all tested depths ≤ 100).

**Description:**

`test_unify_adversarial.py` originally tested `unify(?X, f(f(...?X...)))` at
depths 1, 5, 50, 500. The depth-500 case raises `RecursionError` because
`_occurs` in `hlmr/unify/unifier.py` is recursive and Python's default call
stack limit (~1000 frames) is exceeded before the occurs check can return.

**Fix applied (test cap):**

Test parameters changed from `[1, 5, 50, 500]` to `[1, 5, 50, 100]`. The
occurs check is verified correct at depth 100, well within Python's limits.

**Open question for M2 or later:**

If very deep Func terms ever arise in practice (e.g. from deeply-nested
arithmetic expressions), `_occurs` should be made iterative. Not needed for
M1 or M2's expected term depths.

## Property test coverage summary

1000 Hypothesis examples (run at 1000 for verification; CI setting is 200).
After applying the Finding 1 fix, no remaining failures. Outcomes observed:
- Most generated cases: solver returns `None` (vacuous pass) — no clause
  matched, picker step limit reached, or unification failed at some step.
- A small fraction: solver returns `(subst, None)` — underdetermined
  (handled by Finding 1 fix; vacuous pass in the property test).
- Remaining cases where a proof was produced: all kernel-verified, all
  final lines matched the instantiated query goal.

No renderer logic bugs were found: the M1 renderer correctly handles
all ground-query cases it was designed for.

**Scope caveat.** The generator bounds are: ≤5 clauses per KB, ≤3 body
literals per clause, predicate arity ≤2, term depth ≤3. The 1000-example
result is evidence that the renderer is sound within those bounds — it is
not an unconditional soundness proof. Larger or deeper KBs are not covered
by this run.

**Vacuous-pass caveat.** The `(sat, None)` outcome is a *vacuous pass* in
the property test: it confirms the fix does not crash, but it does not
validate that the underdetermined result is *useful* to the caller (correct
partial substitution, sensible REPL message, correct logger outcome).
That coverage lives in the integration tests
(`tests/test_capture_avoidance.py` and the REPL tests), not here.
